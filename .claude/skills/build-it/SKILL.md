# build-it — ヒアリング駆動実装スキル

ふわっとした課題を持つユーザーから、4フェーズで「課題理解 → 提案 → 合意 → 実装」まで一貫して完了させる。

## 4フェーズ

| Phase | 名前 | 完了条件 |
|-------|------|----------|
| 1. Discovery | 課題ヒアリング | 課題・ユーザー・ゴール・スコープが明確 |
| 2. Proposal | 解決策提案 | ユーザーがアプローチを選択 |
| 3. Contract | 設計合意 | 実装契約を承認 |
| 4. Execution | 実装 | 全検証通過（test, lint, typecheck） |

---

## Phase 1: Discovery（課題ヒアリング）

**目的**: 技術の話からではなく、課題起点で深掘りする。

**ルール**:
- **1回に1つだけ質問する**（複数質問を一度に投げない）
- 各質問に推奨回答（「例えば〜」）を添えて答えやすくする
- コードベースで答えられることは自分で調べて、ユーザーに聞かない
- 相槌や要約を挟みながら自然な会話で進める

**ヒアリングの順序**（上から順に、必要な深さまで掘る）:

1. **課題** — 何に困っている？今どうやっている？何がつらい？
2. **誰が** — 誰が使う？どんな場面で？頻度は？
3. **ゴール** — 理想の状態は？それが実現したら何が変わる？
4. **スコープ** — 今回で解決したい範囲は？後でいいものは？

**進め方**:
- ユーザーの回答から次に深掘りすべき点を判断する
- 十分に理解できたトピックはスキップしてよい
- 全トピックが明確になったら、理解を要約してPhase 2へ進む旨を伝える

---

## Phase 2: Proposal（解決策提案）

**目的**: 課題理解に基づき、コードベースを調査した上で複数のアプローチを提示する。

**手順**:
1. コードベースをresearchエージェントで調査（関連する既存コード・パターンの把握）
2. **2〜3つの解決アプローチ**を提示

**各案のフォーマット**:
```
### 案N: [名前]
[アプローチの概要 — 1-2文、非技術者にもわかる言葉で]

- メリット: ...
- デメリット: ...
- 実装の大きさ: 小 / 中 / 大
- おすすめ度: ★☆☆ / ★★☆ / ★★★
```

**ルール**:
- 必ず1つを **「おすすめ」** として明示し、理由を添える
- 技術的詳細より「何ができるか・何が犠牲になるか」で説明する
- ユーザーが案を選んだら、必要に応じて振る舞いの詳細（エッジケース等）を追加ヒアリングする

---

## Phase 3: Contract（設計合意）

**目的**: 選ばれた案を構造化し、実装前の最終合意を取る。

以下の形式で提示:

```
## 実装契約

### 解決する課題
[1-2文]

### 誰がどう使うか
[ユーザーストーリー形式]

### 選んだアプローチ
[概要]

### スコープ
- **IN**: ...
- **OUT**: ...

### 振る舞い
- 正常系: ...
- エラー系: ...

### ファイル構成
[作成/変更するファイル一覧]

### テスト方針
[主要なテストケース一覧]
```

**ルール**:
- 承認されたら Phase 4 へ進む
- 修正要望があれば該当フェーズまで戻る（Phase 1まで戻ることもある）
- 「OK」「いいよ」「進めて」等の肯定で承認とみなす

---

## Phase 4: Execution（実装）

**目的**: Contract通りに実装し、全検証を通過させる。

**ルール**:
- CLAUDE.mdの開発ルールに完全に従う（TDD、lint、検証等）
- サブエージェントを活用して並列実装する
- 以下が**全て通過するまで完了と言わない**:
  - テスト実行
  - lint
  - typecheck
  - 必要に応じてビルド・ブラウザ確認
- 完了時にContractとの差分があれば報告する

---

## 全体を通じたルール

- 各フェーズの境界で現在のフェーズを明示する（例: 「**[Discovery]** では〜」）
- ユーザーが「戻りたい」と言ったら該当フェーズに戻る
- ユーザーが最初から明確な要件を持っている場合、Discoveryを短縮してよい
- 質問には必ず `AskUserQuestion` ツールを使う
